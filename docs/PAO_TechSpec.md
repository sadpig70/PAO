# PAO — Poly Agents Orchestra

## 파일 메시지 기반 이종 CLI 에이전트 런타임 오케스트레이션 시스템 기술서

| 항목 | 내용 |
|---|---|
| 시스템명 | PAO |
| 정식 명칭 | Poly Agents Orchestra |
| 시스템 유형 | 이종 CLI 에이전트 런타임 오케스트레이션 시스템 |
| 현재 메시지 전송 방식 | 로컬 파일 메시지 버스 |
| 향후 전송 방식 | MCP, 로컬 IPC, SQLite 큐, Redis 등으로 교체 가능 |
| 기본 실행 환경 | 로컬 워크스페이스 |
| 문서 버전 | 0.2 — ADP architecture |
| 작성일 | 2026-07-14 |

---

## 1. 개요

**PAO(Poly Agents Orchestra)**는 서로 다른 벤더, 모델, CLI, TUI 및 헤드리스 실행 방식을 가진 AI 에이전트 런타임들을 하나의 상위 오케스트레이션 계층에서 협업시키는 시스템이다.

PAO의 핵심 목적은 여러 AI 런타임을 단순히 동시에 호출하는 것이 아니다. 상위 오케스트레이터가 목표를 분석하고 작업을 분해한 뒤, 각 작업에 적합한 런타임을 선택하여 지시하고, 실행 결과를 검증·통합하며, 실패하면 재시도하거나 다른 런타임으로 재위임하는 지속적인 협업 루프를 구성하는 것이다.

초기 구현은 별도의 중앙 메시지 브로커나 복잡한 통합 프로토콜 없이 로컬 파일 시스템을 메시지 전달 계층으로 사용한다. 오케스트레이터와 런타임 워커는 약속된 디렉터리, 파일명 및 JSON 스키마를 통해 작업과 결과를 교환한다.

PAO 0.2에서 상위 오케스트레이터는 **OA(Orchestration Agent)**, 실행 runtime은 **LWAR**, LWAR 세션 내부의 상주 자기감시 루프는 **ADP(Agent Daemon Process)**로 정의한다. OA는 LWAR를 비대화형 명령으로 실행하지 않는다. 사용자가 시작한 장기 LWAR 세션이 Python watcher를 반복 호출하여 자기 mailbox를 수신한다.

그러나 **파일 기반 메시징은 PAO의 정체성이 아니라 최초 구현을 위한 전송 방식**이다. PAO의 핵심은 이종 에이전트 런타임의 역할 분담과 협업 오케스트레이션에 있으며, 향후 파일 메시지 버스는 MCP, 로컬 IPC, SQLite, Redis 또는 다른 메시지 전송 계층으로 교체할 수 있다.

## 2. 명칭과 시스템 정체성

### 2.1 정식 명칭

> **PAO — Poly Agents Orchestra**

`Poly`는 서로 다른 벤더와 실행 특성을 가진 다양한 에이전트를 의미하고, `Agents`는 독립적인 추론 및 작업 수행 주체들을 의미한다. `Orchestra`는 상위 오케스트레이터의 지휘 아래 각 에이전트가 서로 다른 역할을 맡고, 부분 결과를 전체 목표로 통합하는 협업 구조를 나타낸다.

### 2.2 권장 설명문

영문:

> **PAO is a local-first orchestration system that coordinates heterogeneous CLI agent runtimes through interchangeable message transports.**

한국어:

> **PAO는 교체 가능한 메시지 전송 계층을 통해 이종 CLI 에이전트 런타임을 협업시키는 로컬 워크스페이스 중심 오케스트레이션 시스템이다.**

### 2.3 PAO가 의미하지 않는 것

PAO는 다음 중 어느 하나에만 한정되지 않는다.

- 특정 AI 모델 또는 벤더의 멀티 에이전트 기능
- 단순한 프롬프트 병렬 호출기
- 파일 동기화 프로그램
- 단일 CLI의 세션 관리자
- 파일 기반 메시징만 지원하는 고정 구조
- 모든 런타임이 동일 모델을 사용하는 동종 에이전트 시스템

## 3. 목표

### 3.1 1차 목표

Claude Code CLI, Codex CLI, Grok 계열 런타임, DeepSeek TUI, Kimi Code CLI, Qwen CLI, OpenCode 등 서로 다른 실행 환경을 공통 작업 계약 아래에서 동작시키는 것이다.

### 3.2 2차 목표

인간의 반복적인 수동 조작 없이 오케스트레이터가 다음 과정을 수행하는 자동 협업 파이프라인을 구축하는 것이다.

1. 목표 해석
2. 작업 분해
3. 런타임 선택
4. 역할 및 제약 부여
5. 작업 전달
6. 실행 결과 회수
7. 결과 검증
8. 실패 복구 또는 재위임
9. 부분 결과 통합
10. 최종 결과와 근거 보고

### 3.3 설계 목표

- 단일 머신과 로컬 파일 시스템만으로 최소 기능을 구현할 수 있어야 한다.
- 중앙 메시지 브로커 없이도 동작할 수 있어야 한다.
- 런타임별 차이는 어댑터 계층에서 흡수해야 한다.
- 긴 단일 세션보다 짧은 실행 슬라이스의 반복을 우선해야 한다.
- 작업과 결과는 런타임에 종속되지 않는 공통 스키마로 표현해야 한다.
- 메시지 전송 계층을 교체해도 오케스트레이터와 런타임 어댑터의 핵심 로직은 유지되어야 한다.
- 실패, 타임아웃, 중복 실행 및 워커 중단을 탐지하고 복구할 수 있어야 한다.

## 4. 핵심 설계 원칙

### 4.1 오케스트레이션과 전송의 분리

PAO는 작업 계획과 메시지 전달을 분리한다.

- **오케스트레이션 평면**: 목표 분해, 런타임 선택, 검증, 재계획 및 결과 통합
- **전송 평면**: 작업 지시, 상태, 결과 및 제어 메시지의 전달
- **실행 평면**: 각 런타임의 실제 추론, 파일 조작, 명령 실행 및 산출물 생성
- **상태 평면**: 태스크 수명주기, 워커 상태, 재시도 횟수 및 실행 이력 관리

초기에는 전송 평면과 상태 평면의 일부를 파일 시스템이 담당한다. 향후 MCP나 데이터베이스 큐를 도입하더라도 오케스트레이션 계약은 유지한다.

### 4.2 공통 작업 계약

모든 런타임은 내부 인터페이스가 다르더라도 PAO 외부에서는 동일한 태스크와 결과 계약을 사용한다. 런타임 고유 옵션은 공통 스키마를 오염시키지 않도록 별도 필드에 격리한다.

### 4.3 짧은 실행의 연속성

워커 프로세스는 상시 대기할 수 있지만 실제 AI 런타임은 작업이 발생할 때만 호출하고 종료하는 것을 기본으로 한다. 장시간 작업은 하나의 영구 세션이 아니라 여러 개의 짧은 작업을 연속적으로 연결하여 수행한다.

### 4.4 검증 없는 성공 금지

런타임이 정상 종료되었다는 사실만으로 태스크 성공을 확정하지 않는다. 요구 산출물, 테스트, 형식, 근거 및 품질 조건을 별도로 검증해야 한다.

### 4.5 런타임 교체 가능성

특정 런타임이 실패하거나 비용·속도·품질 조건에 맞지 않으면 동일 작업을 다른 런타임으로 이전할 수 있어야 한다. 이를 위해 작업 지시는 가능한 한 벤더 중립적으로 작성한다.

## 5. 시스템 범위

### 5.1 포함 범위

- 다단계 작업 계획 및 태스크 그래프 생성
- 런타임별 역할 할당
- 메시지 전달 및 결과 회수
- 로컬 CLI, TUI, 헤드리스 런타임 실행
- 상태, heartbeat, lock 및 로그 관리
- 결과 정규화
- 결과 검증 및 교차 검토
- 타임아웃, 재시도, 대체 런타임 위임
- 파일, 패치, 보고서 등 산출물 추적
- 비용, 지연 시간 및 성공률 기반 라우팅 확장

### 5.2 초기 제외 범위

- 밀리초 단위 실시간 메시지 처리
- 대규모 글로벌 분산 트랜잭션
- 모델 자체의 학습 또는 파인튜닝
- 런타임 벤더의 인증·과금 시스템 대체
- 신뢰되지 않은 임의 코드를 완전하게 격리하는 범용 보안 샌드박스
- 모든 TUI 프로그램에 대한 보편적 자동화 보장

## 6. 상위 아키텍처

PAO는 다음 여섯 개의 논리 계층으로 구성한다.

```text
┌─────────────────────────────────────────────┐
│ 1. PAO Orchestrator                         │
│ 목표 분석 · 태스크 분해 · 라우팅 · 재계획  │
├─────────────────────────────────────────────┤
│ 2. Validation & Synthesis Layer             │
│ 결과 검증 · 교차 검토 · 통합 · 최종 판정   │
├─────────────────────────────────────────────┤
│ 3. Task & State Layer                       │
│ 태스크 DAG · 상태 머신 · 재시도 · 이력      │
├─────────────────────────────────────────────┤
│ 4. Message Transport Interface              │
│ FileBus / MCP / SQLite / Redis / IPC        │
├─────────────────────────────────────────────┤
│ 5. Runtime Workers & Adapters               │
│ 감시 · 변환 · 실행 · 정규화 · 정리          │
├─────────────────────────────────────────────┤
│ 6. Poly Agent Runtimes                      │
│ Claude · Codex · Qwen · Kimi · Grok · GLM  │
└─────────────────────────────────────────────┘
```

### 6.1 PAO Orchestrator

상위 목표를 태스크 그래프로 분해하고 어떤 런타임에 어떤 역할을 맡길지 결정한다. 단순 라우터가 아니라 시스템 전체의 추론과 제어를 담당한다.

주요 책임은 다음과 같다.

- 목표와 완료 조건 해석
- 선행 관계를 포함한 서브태스크 생성
- 런타임 능력, 가용성, 비용 및 위험도 평가
- 태스크 우선순위 및 타임박스 결정
- 결과 품질 판정
- 재시도, 재계획 및 대체 런타임 선택
- 최종 산출물 통합

### 6.2 Validation & Synthesis Layer

결과가 작업 계약을 충족했는지 확인한다. 필요한 경우 다른 런타임이 결과를 교차 검토하게 하며, 상충하는 결과가 발생하면 근거 비교 또는 추가 검증 태스크를 생성한다.

### 6.3 Task & State Layer

태스크의 수명주기와 관계를 관리한다. MVP에서는 JSON 파일과 디렉터리 이동으로 구현하고, 작업량이 증가하면 SQLite를 병행할 수 있다.

### 6.4 Message Transport Interface

오케스트레이터와 워커 사이에서 메시지를 전달하는 추상 인터페이스다. 초기 구현체는 `FileTransport`이며 향후 `MCPTransport` 등으로 교체할 수 있다.

### 6.5 Runtime Workers & Adapters

워커는 메시지를 수신하고 담당 런타임을 호출한다. 어댑터는 공통 태스크를 각 런타임의 명령행 인자, 프롬프트 및 입출력 형식으로 변환한다.

### 6.6 Poly Agent Runtimes

실제 추론과 작업을 수행하는 이종 AI 런타임 집합이다. 각 런타임은 서로 다른 모델, 컨텍스트 정책, 도구, 권한 및 사용자 인터페이스를 가질 수 있다.

## 7. 전송 계층 추상화

### 7.1 공통 인터페이스

전송 계층은 최소한 다음 기능을 제공해야 한다.

```python
class MessageTransport:
    def publish_task(self, runtime_id, task): ...
    def lease_task(self, runtime_id, worker_id): ...
    def publish_result(self, result): ...
    def read_result(self, task_id): ...
    def publish_heartbeat(self, worker_state): ...
    def read_heartbeat(self, runtime_id): ...
    def acknowledge(self, task_id, worker_id): ...
    def reject(self, task_id, reason): ...
```

구현체는 파일, MCP, 데이터베이스 또는 메시지 브로커를 사용할 수 있지만 상위 오케스트레이터에는 동일한 의미를 제공해야 한다.

### 7.2 초기 FileTransport

초기 버전은 로컬 파일 시스템을 전송 매체로 사용한다. 장점은 다음과 같다.

- 구현 및 디버깅이 단순하다.
- 메시지와 결과를 사람이 직접 열어볼 수 있다.
- 별도 서버나 브로커가 필요 없다.
- 실행 이력과 산출물이 자연스럽게 남는다.
- 단일 워크스테이션 환경에서 충분한 성능을 제공한다.

### 7.3 향후 MCPTransport

MCP 전환 시에도 태스크 및 결과 스키마를 유지한다. 변경되는 부분은 메시지 전달과 도구 호출 바인딩이며, 오케스트레이터의 작업 분해·검증·재계획 로직은 유지하는 것을 원칙으로 한다.

## 8. 파일 기반 MVP 디렉터리 구조

```text
pao/
  config/
    runtimes.json
    routing.json
    policies.json
  inbox/
    claude_code.json
    codex.json
    qwen.json
    kimi.json
  outbox/
    claude_code.json
    codex.json
    qwen.json
    kimi.json
  heartbeat/
    claude_code.json
    codex.json
    qwen.json
    kimi.json
  lock/
    claude_code.lock
    codex.lock
    qwen.lock
    kimi.lock
  tasks/
    pending/
    leased/
    running/
    retry_wait/
    done/
    failed/
    dead_letter/
  results/
  artifacts/
  logs/
  archive/
```

### 8.1 단일 파일 모델

`inbox/{runtime}.json`과 `outbox/{runtime}.json`을 사용하는 방식이다. 구현이 가장 단순하므로 2~3개 런타임을 통합하는 최초 MVP에 적합하다.

### 8.2 큐 디렉터리 모델

병렬성과 다중 작업 큐잉이 필요해지면 태스크별 파일을 생성하고 상태에 따라 디렉터리를 이동한다.

```text
tasks/pending/{priority}_{task_id}.json
tasks/running/{task_id}.json
tasks/done/{task_id}.json
```

태스크 파일 하나가 하나의 불변 작업 계약을 나타내며, 상태 변경은 메타데이터 갱신 또는 원자적 디렉터리 이동으로 표현한다.

## 9. 파일 규약

| 경로 | 의미 | 작성 주체 | 읽기 주체 |
|---|---|---|---|
| `inbox/{runtime}.json` | 대상 런타임의 최신 지시 | 오케스트레이터 | 워커 |
| `outbox/{runtime}.json` | 대상 런타임의 최신 결과 | 워커 | 오케스트레이터 |
| `heartbeat/{runtime}.json` | 워커 생존 및 가용 상태 | 워커 | 오케스트레이터 |
| `lock/{runtime}.lock` | 중복 실행 방지 | 워커 | 워커·오케스트레이터 |
| `tasks/*/{task_id}.json` | 태스크 상태 및 계약 | 양측 | 양측 |
| `results/{task_id}.json` | 정규화된 실행 결과 | 워커 | 오케스트레이터·검증기 |
| `artifacts/{task_id}/` | 패치·보고서·출력 파일 | 워커 | 전체 계층 |
| `logs/{runtime}.log` | 표준·오류·감사 로그 | 워커 | 운영자·오케스트레이터 |

파일명과 디렉터리 구조는 전송 프로토콜의 일부이므로 버전 관리해야 한다. 임의 변경 대신 스키마 버전과 마이그레이션 정책을 사용한다.

## 12. ADP 기반 LWAR 실행 설계

각 LWAR는 사용자가 대화형 또는 TUI 방식으로 한 번 실행한다. LWAR는 `lwar-runtime` 스킬을 로드한 뒤 자신의 세션 안에서 ADP를 유지한다. Python watcher는 메시지 I/O를 결정론적으로 수행하지만 반복과 Task 수행의 주체는 LWAR Agent 자신이다.

### 12.1 기본 루프

1. LWAR 세션 시작과 자기등록
2. OA의 `LWARn` identity 승인
3. ADP watcher Python 코드 실행
4. watcher가 1초 간격으로 자기 mailbox 검사
5. 메시지가 없으면 90초 후 `idle_timeout`을 stdout으로 출력하고 종료
6. LWAR가 같은 watcher를 즉시 재실행
7. Task가 있으면 `incoming → claimed` 원자 이동 후 Task를 stdout으로 출력
8. LWAR가 같은 세션에서 Task 수행
9. 결과를 `outgoing`에 원자 저장
10. LWAR가 watcher를 재실행
11. `shutdown` control을 받을 때만 ADP 종료

### 12.2 Watcher와 LWAR 세션의 분리

90초마다 종료되는 것은 Python watcher뿐이다. LWAR Agent 세션은 종료되지 않으며 watcher stdout을 다음 행동 입력으로 사용한다.

이 방식은 다음 효과가 있다.

- 비대화형 명령을 지원하지 않는 TUI runtime 수용
- 기존 대화 컨텍스트와 runtime 고유 도구 유지
- OA의 vendor별 실행 로직 제거
- 공통 mailbox·Skill만으로 이종 runtime 통합
- watcher timeout을 통한 heartbeat와 제어 확인

반복되는 idle event는 최소 JSON sentinel로 유지하여 LWAR context 오염을 제한한다.

## 13. ADP 결정론적 도구

모든 LWAR는 동일한 Python 도구 계약을 사용한다.

```python
class ADPTools:
    def register(self, runtime_profile): ...
    def watch(self, identity, interval_s=1, timeout_s=90): ...
    def claim(self, message_file): ...
    def heartbeat(self, identity, state): ...
    def submit_result(self, identity, task_id, result): ...
```

### 13.1 LWAR 상주 세션 책임

- 자기 runtime profile 등록과 승인된 `LWARn` identity 채택
- 1초 polling, 90초 timeout의 watcher slice 반복 실행
- stdout ADP event 해석과 Task 수행
- 권한 정책과 완료 조건 준수
- 공통 Result JSON 작성과 원자 제출
- `idle_timeout` 후 재감시, `shutdown` 후 종료
- heartbeat와 lease를 통한 생존·수행 상태 공개
- 자기 mailbox 이외 경로의 직접 조작 금지

### 13.2 능력 프로필

각 런타임은 다음과 같은 능력 프로필을 제공할 수 있다.

## 14. 태스크 상태 머신

최소 상태는 다음과 같다.

| 상태 | 의미 |
|---|---|
| `pending` | 실행 대기 |
| `leased` | 특정 워커가 제한 시간 동안 작업권 확보 |
| `running` | 런타임 실행 중 |
| `succeeded` | 실행 및 검증 성공 |
| `failed` | 실행 또는 작업 실패 |
| `retry_wait` | 재시도 지연 중 |
| `cancelled` | 상위 정책이나 사용자에 의해 취소 |
| `dead_letter` | 자동 복구 한도를 초과하여 격리 |

권장 전이는 다음과 같다.

```text
pending → leased → running → succeeded
                    │
                    └→ failed → retry_wait → pending
                                      │
                                      └→ dead_letter
```

`succeeded`는 런타임 종료 성공과 검증 성공을 모두 만족한 상태로 정의한다. 실행은 성공했지만 검증에 실패한 경우 `failed` 또는 별도의 `validation_failed` 상태를 사용할 수 있다.

## 15. 락, lease 및 원자적 쓰기

파일 기반 MVP에서 가장 중요한 안정성 문제는 부분 쓰기와 중복 실행이다.

### 15.1 원자적 쓰기

1. 동일 디렉터리에 임시 파일 작성
2. 파일 내용 flush
3. 필요 시 파일 시스템 동기화
4. rename 또는 replace로 최종 파일 교체

### 15.2 Lock

- 작업 실행 전 lock 생성
- lock에 `worker_id`, `task_id`, `created_at`, `expires_at` 기록
- 정상 종료 또는 실패 처리 후 lock 해제
- 만료된 lock은 stale lock으로 분류
- stale lock 회수는 감사 로그를 남긴 뒤 수행

### 15.3 Lease

단순 lock보다 태스크 lease를 권장한다. 워커는 제한 시간 동안만 실행권을 보유하며, heartbeat 또는 lease 갱신이 중단되면 오케스트레이터가 작업을 회수할 수 있다.

PAO의 기본 전달 보장은 `at-least-once`로 설계할 수 있다. 따라서 어댑터와 작업은 가능한 한 멱등성을 가져야 하며, 동일 `task_id`의 결과가 이미 존재하면 중복 실행 여부를 확인해야 한다.

## 17. 오케스트레이션 전략

### 17.1 역할 기반 라우팅

예시 역할은 다음과 같다.

| 역할 | 주요 작업 |
|---|---|
| `planner` | 목표 분해, 태스크 DAG 생성 |
| `researcher` | 자료 조사, 근거 수집 |
| `coder` | 구현 및 수정 |
| `tester` | 테스트 작성과 실행 |
| `reviewer` | 코드·논리·보안 검토 |
| `verifier` | 완료 조건과 증거 검증 |
| `synthesizer` | 부분 결과 통합 |

역할은 특정 런타임에 고정하지 않는다. 런타임의 능력, 현재 부하, 비용 및 과거 성공률에 따라 동적으로 매핑한다.

### 17.2 정책 기반 선택

런타임 선택 점수는 다음 요소를 조합할 수 있다.

- 태스크 적합도
- 최근 성공률
- 평균 처리 시간
- 비용 등급
- 컨텍스트 요구량
- 필요한 도구와 권한
- 현재 워커 상태
- 독립 검증 필요성

### 17.3 교차 검증

중요 태스크는 작성 런타임과 검증 런타임을 분리한다. 동일한 오류나 편향이 반복되지 않도록 가능하면 서로 다른 모델 계열을 사용한다.

### 17.4 충돌 해결

서로 다른 런타임 결과가 충돌하면 단순 다수결로 결정하지 않는다. 검증 가능한 증거, 테스트 결과, 출처, 재현성 및 완료 조건을 기준으로 판정한다. 판정이 불가능하면 추가 검증 태스크를 생성하거나 사람 게이트로 전환한다.

## 18. 오류 처리와 재시도

### 18.1 오류 분류

| 오류 유형 | 예시 |
|---|---|
| 런타임 오류 | 실행 실패, 인증 오류, 비정상 종료 |
| 작업 오류 | 출력은 있으나 목표 미달성 |
| 검증 오류 | 테스트 실패, 산출물 누락, 근거 부족 |
| 전송 오류 | 파일 손상, 메시지 읽기 실패 |
| 상태 오류 | lock 충돌, heartbeat 중단, 중복 lease |
| 정책 오류 | 권한 위반, 허용되지 않은 명령 |

### 18.2 재시도 정책

- 일시적 오류는 지수 백오프로 동일 런타임 재시도
- 동일 원인으로 반복 실패하면 다른 런타임으로 재위임
- 작업 범위가 과도하면 더 작은 태스크로 재분해
- 최대 재시도 횟수를 초과하면 `dead_letter`로 이동
- 보안 또는 권한 위반은 자동 재시도하지 않음
- 모든 재시도는 이전 실패 원인과 변경된 전략을 기록

## 19. 보안 모델

PAO는 여러 런타임이 로컬 워크스페이스와 명령 셸에 접근할 수 있으므로 최소 권한 원칙을 적용해야 한다.

- 태스크별 읽기·쓰기 허용 경로 지정
- 실행 가능한 명령과 도구 제한
- 네트워크 사용 여부를 태스크 정책으로 통제

### 19.2 신뢰 경계

AI 런타임의 자연어 출력은 신뢰된 제어 명령으로 취급하지 않는다. 워커는 구조화된 결과 필드와 명시적인 정책을 기준으로 후속 행동을 수행해야 한다.

### 19.3 감사 가능성

태스크 생성, 배정, 실행 명령, 권한 승인, 결과, 검증 및 재시도 이력을 상호 연결 가능한 ID와 함께 기록한다.

## 22. 파일 메시징에서 MCP로의 전환 전략

PAO의 명칭에서 파일을 제외한 이유는 전송 기술을 시스템 정체성과 분리하기 위해서다.

### 22.1 유지해야 하는 요소

- `task_id`, `workflow_id` 및 상관관계 ID
- 공통 태스크 스키마
- 공통 결과 스키마
- 상태 머신 의미
- 런타임 어댑터 인터페이스
- 검증 및 재시도 정책
- 권한과 감사 모델

### 22.2 교체되는 요소

- 파일 경로 기반 publish/read
- 디렉터리 이동 기반 상태 변경
- lock 파일
- heartbeat 파일
- 폴링 또는 파일 변경 감시

### 22.3 MCP 전환 원칙

- `FileTransport`와 `MCPTransport`가 동일한 상위 계약을 구현해야 한다.
- 전환 기간에는 두 전송을 동시에 지원할 수 있어야 한다.
- 런타임별로 서로 다른 전송 방식을 사용할 수 있어야 한다.
- 기존 파일 메시지를 MCP 요청으로 변환하는 브리지 구현을 허용한다.
- 메시지 의미와 전달 구현을 혼합하지 않는다.

이를 통해 초기 파일 기반 자산을 폐기하지 않고 단계적으로 전환할 수 있다.

## 23. 구현 단계

### 23.1 1단계 — PAO MVP

- OA 1개
- `oa-runtime`·`lwar-runtime` 스킬
- 동적 LWAR 자기등록
- ADP watcher와 결과 제출 도구
- `FileTransport` 구현
- `mailbox/LWARn/{incoming,claimed,outgoing}` 모델
- 공통 태스크 및 결과 스키마
- heartbeat, lock, 로그
- 1초 polling·90초 watch slice
- 단순 성공·실패·재시도 루프

### 23.2 2단계 — 안정화

- 태스크 상태 머신
- lease와 stale lock 회수
- 오류 분류와 재시도 정책
- 결과 검증 런타임
- 원자적 쓰기와 중복 처리 방지
- 런타임 능력 프로필
- 기본 운영 지표

### 23.3 3단계 — 다중 런타임 확장

- Qwen, Kimi, Grok, DeepSeek, OpenCode의 ADP skill 채택
- 태스크별 큐 디렉터리
- 병렬 실행과 우선순위 스케줄링
- 비용·속도·성공률 기반 라우팅
- 다중 런타임 교차 검증
- SQLite 상태 저장소 병행

### 23.4 4단계 — 전송 계층 확장

- `MessageTransport` 인터페이스 확정
- `MCPTransport` 구현
- File–MCP 브리지
- 런타임별 전송 선택
- 다중 호스트 워커 지원 검토

## 24. 적합한 활용 시나리오

### 24.1 소프트웨어 개발

한 런타임이 요구사항을 분석하고, 다른 런타임이 구현하며, 또 다른 런타임이 테스트와 리뷰를 수행하는 파이프라인을 구성할 수 있다.

### 24.2 장시간 연구

리서치, 근거 검증, 반론 생성, 종합 보고서 작성을 독립 태스크로 나누어 야간 배치로 수행할 수 있다.

### 24.3 다중 모델 검증

동일 문제를 서로 다른 런타임에 독립적으로 할당한 뒤 근거와 결과를 비교하여 단일 모델의 오류 가능성을 낮출 수 있다.

### 24.4 반복형 개선 루프

생성 → 테스트 → 오류 분석 → 수정 → 재검증 작업을 완료 조건이 충족될 때까지 반복할 수 있다.

### 24.5 역할 특화 협업

런타임별 강점을 측정해 계획, 코딩, 분석, 검증 및 문서화 역할을 동적으로 배정할 수 있다.

## 25. 한계

- 파일 기반 전송은 높은 동시성과 대규모 분산 처리에 한계가 있다.
- runtime 세션이 shell/Python 실행 또는 장기 대화를 유지하지 못하면 ADP에 참여할 수 없다.
- runtime이 ADP stdout 계약을 따르지 않으면 OA가 작업 완료를 보장할 수 없다.
- 여러 모델이 동일한 잘못된 전제를 공유할 경우 교차 검증만으로 오류가 제거되지 않을 수 있다.
- 로컬 명령 실행 권한은 보안 위험을 수반하므로 정책과 격리가 필수다.
- 자동 오케스트레이션의 품질은 태스크 분해와 완료 조건의 품질에 크게 의존한다.

## 27. 결론

**PAO(Poly Agents Orchestra)**는 서로 다른 AI 에이전트 런타임을 하나의 협업 체계로 조직하는 상위 오케스트레이션 시스템이다. 각 런타임은 독립적인 악기처럼 고유한 모델, 도구, 인터페이스 및 강점을 가지며, PAO Orchestrator는 전체 목표를 기준으로 역할을 분배하고 결과를 하나의 성과로 통합한다.

초기 파일 메시지 방식은 PAO를 빠르게 구현하고 검증하기 위한 현실적인 선택이다. 파일은 사람이 확인할 수 있고 별도 인프라가 필요 없으며 로컬 워크스페이스 자동화에 적합하다. 동시에 전송 계층을 추상화함으로써, 시스템이 성장하면 MCP나 다른 메시지 기술로 단계적으로 전환할 수 있다.

PAO의 성공 여부는 특정 메시지 기술보다 다음 요소에 의해 결정된다.

1. 명확한 공통 작업 계약
2. 정규화된 결과와 검증 가능한 완료 조건
3. 런타임별 어댑터의 안정성
4. heartbeat, lease 및 실패 복구 정책
5. 런타임 강점에 맞는 역할 배정
6. 검증과 재조정을 포함한 지속적인 오케스트레이션 루프

이 원칙을 유지하면 PAO는 파일 기반 MVP에서 시작해 MCP와 다중 호스트 실행을 지원하는 확장형 이종 AI 에이전트 협업 플랫폼으로 발전할 수 있다.
